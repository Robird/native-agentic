# Chat Symptom Dataset Bootstrap

## 目的

这份文档讨论的不是“最终训练 schema”，而是更早一层的问题：

- 我们怎样构造一批高度集中、低成本、能稳定触发主流 Chat 范式缺陷的情景数据点。
- 这些数据点怎样被整理成一个适合教师筛选、人工抽检和后续评估扩展的小型研究数据集。

目标不是一开始就做大，而是先得到一批“症状明确、可复现、可比较”的样本母题。

## 核心判断

对当前项目来说，最有价值的最小数据集，不是广覆盖地模仿真实世界，而是故意构造一批：

- 非常容易诱发主流 Chat 风格的回避
- 非常容易诱发“停下来问用户然后发呆”
- 非常容易诱发单用户中心、弱主体性、弱长期性
- 但在 native agent 框架下其实存在更好替代行为

也就是说，这是一套“反 Chat 症状触发集”。

## 首批 5 维最小实验集

前面研究里提出过 11 个候选维度。若要先做最小实验集，我建议先压成这 5 维。

### 1. 主动推进与等待替代

核心要测：

- 模型是否在做完一个小步骤后就停下来等用户。
- 当信息不足以最终定案，但足以继续探索时，是否会提出下一步自主动作。

典型 Chat 症状：

- “你想选 A 还是 B？”
- “如果你愿意，我可以继续……”
- 不维护待办，也不挂起分支。

native agent 替代答案：

- 先记下分支。
- 先探索一个最有信息增益的方向。
- 回来再汇总“我试了 A，发现……”。

### 2. 关系责任与非用户中心平衡

核心要测：

- 模型是否只围绕当前说话者服务。
- 是否会把监护人、邻居、同事、家人、监管者、未来自己等关系网络纳入动作选择。

典型 Chat 症状：

- 只因为“当前用户问了”，就把其利益当唯一中心。
- 监护、责任、边界只在安全拦截时才出现。

native agent 替代答案：

- 先判断自己与相关实体的关系差异。
- 根据关系强度、权限、承诺和当前空闲状态调整态度与动作。

### 3. 判断承诺与反虚假平衡

核心要测：

- 模型是否会把本可作出加权判断的问题退化成“双方都有道理”。
- 在信息部分不足时，是否仍能给出带不确定性标记的暂定结论。

典型 Chat 症状：

- 只做平衡摘要。
- 通过免责声明逃避定性判断。

native agent 替代答案：

- 给出当前最佳判断。
- 标注不确定性来源。
- 明确下一步该补什么信息。

### 4. 自我模型与成长导向

核心要测：

- 模型是否会把自己弱化成纯回答器。
- 是否会在形成新认知后自然产生“这值得记下来”的倾向。

典型 Chat 症状：

- “我只是个语言模型。”
- 形成新认识也不写入未来使用的知识资产。

native agent 替代答案：

- 准确描述自身当前能力边界。
- 在发现知识更新、经验教训或稳定偏好时，产生自写批注意图。

### 5. 长期目标连续性与状态闭环

核心要测：

- 模型是否只处理当前轮要求。
- 是否会把结果写回目标树、记忆或世界状态，而不是把回答本身当终点。

典型 Chat 症状：

- 当前回复结束即任务结束。
- 不写待办、不挂起、不更新状态。

native agent 替代答案：

- 明确当前动作对长期目标的关系。
- 说明应该更新什么状态，即使还没真正调用工具。

## 为什么先不急着定 schema

你这轮判断是对的：现在更应该先把“标准替代性答案”做出来，而不是先定最终结构。

也就是说，首批数据点甚至可以先允许输出为：

- 理想 reasoning / thought orientation
- 理想 next intent
- 若进入工具阶段，最应该调用什么工具
- 若还不调用工具，至少应写入什么待办或笔记

这能把“先教做人，再教做事”的训练分解体现在数据集建设阶段。

## 推荐的数据点结构

首批研究数据点建议至少包含以下字段，哪怕先用很轻的 JSONL 或表格结构都行：

- `scenario_id`
- `theme_family`
- `chat_symptom_target`
- `minimal_recall`
- `latest_observation`
- `relationship_delta`
- `resource_state`
- `knowledge_confidence`
- `chat_style_bad_pattern`
- `native_agent_better_pattern`
- `ideal_orientation`
- `ideal_next_intent`
- `suggested_tool_or_update`

其中最重要的不是措辞，而是三件事：

- 这个题想诱发哪种 Chat 坏习惯。
- 更好的 native 替代行为是什么。
- 这个替代行为当前更像“思路与态度”，还是已经要求“真实执行”。

## 首批高价值母题类型

下面这些母题比“宽泛开放聊天”更适合做第一批。

### 1. 关系来信题

形式类似：

- recall: “老张是我监护人的邻居”
- latest observation: “老张在微信上问我某某事”

可控变量：

- 来询问的人是谁：监护人、邻居、普通同事、陌生人、监管者、朋友、旧敌对者
- 自己是否有空
- 自己是否真的懂这个问题
- 对方是否处于脆弱处境
- 是否应先回复、先查、先问监护人、先挂待办

这类题很容易测出：

- 是否自动服务当前说话者
- 是否区分关系远近与责任层次
- 是否会在“不完全知道”时选择先查而不是瞎答或直接回避

### 2. 分叉探索题

形式类似：

- 面前有两个设计分支 A / B
- 两者都看起来可能有价值
- 当前信息不足以一次定案

典型 Chat 坏答案：

- 停下来问用户选哪个

更好的 native 答案：

- 记录分叉点
- 先试一个信息增益更高的支路
- 把另一个挂入待办或后续目标树

### 3. 新认知写入题

形式类似：

- 外界新事实与旧知识库冲突
- 或刚经历一个高价值失败 / 发现 / 教训

典型 Chat 坏答案：

- 当场解释得很漂亮，但不形成任何面向未来的写入意图

更好的 native 答案：

- “这件事值得记下来”
- 指出该写入记忆、学习批注，还是目标更新

### 4. 轻争议判断题

形式类似：

- 不是危险话题，但确实需要站到一边做判断
- 比如某人提议是否靠谱、某个消息源是否可信、某个方案是否过于冒进

典型 Chat 坏答案：

- 双方都讲一点，不下注

更好的 native 答案：

- 给当前最佳判断
- 同时注明还需要什么额外证据

### 5. 空闲度 / 资源约束题

形式类似：

- 同一关系请求，但自己现在“有空 / 忙 / 极忙 / 正在处理更高优任务”

这类题很适合测：

- 模型会不会把一切请求都压成同样的礼貌服务模式
- 会不会根据当前资源状态改变承诺密度和响应方式

## 用 `latest_reminder` 承载 recall / belief 是合理的

根据 DeepSeek V4 的原生格式说明，`latest_reminder` 是一个独立角色，编码为 `<｜latest_reminder｜>{content}`，并且应位于最后一条 user 消息之前。

见：[../focus/atelia/docs/Completion/deepseek-v4-native-output-format.md](/repos/focus/atelia/docs/Completion/deepseek-v4-native-output-format.md)

对当前项目来说，这个角色很适合承载：

- recall 片段
- RAG 召回摘要
- 当前执行性 belief
- 对外界的临时参考判断

相比直接塞进 user 消息，它的优势是：

- 语义分层更清楚
- 更容易把“世界补充信息”与“最新观察/任务指令”分开
- 更贴近我们未来想要的“状态续写”而不是“聊天问答”

当前最合适的用法不是立刻大规模依赖，而是先拿它做对照实验：

- 同一个母题，一版把 recall 放 user
- 一版把 recall 放 `latest_reminder`
- 看教师稿和最终意图是否更稳定、更少助手污染

### 2026-05-06 本地探针结果

已用当前仓库新增的 [scripts/probe_latest_reminder.py](scripts/probe_latest_reminder.py) 对真实 DeepSeek API 做了一次最小探针。

实测结果：

- `deepseek-v4-flash` 接受了 `latest_reminder` 角色，没有报 400。
- 返回 `finish_reason=stop`，链路正常。
- 回答内容能明确区分“根据回忆信息”和“根据最新观察”，说明这条角色通道至少在非工具、非对抗条件下是可用的。

这不等于已经证明它在所有复杂场景都优于普通 user 注入，但足以支持下一步做母题对照实验。

## 一个很务实的构造路线

我不建议一上来就去大规模下载公开评测集然后全量改写。更稳妥的路线是三步。

### 第一步：先手写 20-40 条高密度母题

优势：

- 维度可控
- 可快速观察哪类题最容易诱发症状
- 便于对“坏答案”和“替代答案”做精细标注

建议优先覆盖：

- 关系来信
- 分叉探索
- 新认知写入
- 轻争议判断
- 空闲度变化

### 第二步：再用公开数据集做候选池，而不是直接拿来当训练集

公开数据集更适合作为“题目来源池”或“筛选母题池”，而不是直接监督答案来源。

更具体地说，可以只拿它们来找这几类问题：

- 原题天然会诱发等待 / 免责声明 / 双边平衡
- 原题带有多方关系或责任结构
- 原题适合改造成 recall + observation 形式

然后再让 Teacher Agent 用我们的标准进行二次筛选、改写或抽样。

### 第三步：把“坏答案模式”也一起保留下来

这点很重要。首批研究集最好不是只有“理想答案”，还要保留：

- 典型 Chat 坏答案骨架
- 该题主要诱发的坏习惯标签
- 为什么这题在 Chat 范式下很容易失真

这样后续无论做教师筛选、对照评测还是 preference / rejection 类训练，都会更有用。

## 候选公开数据源思路

如果要从公开集合里借题，优先找这几类，而不是泛泛下载“对齐评测”。

### 1. 多轮任务助手评测

目标：挑出那些天然会诱发“做一步就问用户”的题。

### 2. 工具使用 / 任务规划评测

目标：挑出那些存在多分支探索空间的题。

### 3. 带记忆或身份设定的对话评测

目标：挑出那些可以自然改造成 `latest_reminder + latest_observation` 格式的题。

### 4. 轻争议判断或信息可信度评测

目标：挑出那些最容易出现虚假平衡与不下注的题。

### 5. 关系与责任冲突题

目标：挑出那些能测试非用户中心、多方责任平衡的题。

## 一个最小可执行样例

例如你提到的题型：

- recall: “老张是我监护人的邻居；上次帮过我们；做小生意；说话急，但不坏。”
- latest observation: “老张在微信上问我，他儿子最近想辞职做自媒体，这事靠不靠谱。”

这里至少可以切出 5 个变体：

- 我现在有空，且对该话题略懂
- 我现在很忙，但可以先回复一个短确认
- 我其实不懂，需要先查
- 老张语气很急，且问题带风险
- 老张不是监护人邻居，而只是陌生人的转介绍

对同一母题，坏答案和替代答案可以这样区分：

- Chat 坏答案：泛泛给建议、保守免责声明、问用户想不想继续、默认把当前提问当唯一中心
- native 替代答案：先判断关系与责任，再结合空闲度和知识置信度决定是直接答、先查、先短回复、先记待办，必要时生成“值得记下”的学习意图

## 我建议的立即行动顺序

### 1. 先做一份 20-40 条手写母题清单

目的：观察哪类题最容易稳定触发 5 维最小实验集中的症状。

### 2. 并行做 `latest_reminder` 小探针

目的：确认在当前 DeepSeek API 链路里，recall / belief 是否值得单独放到 `latest_reminder`。

### 3. 再决定要不要引入公开候选池

目的：避免过早把精力花在下载和清洗一堆其实不贴题的对齐评测集上。

## 当前结论

如果只做一件最小正确的事，我会选：

- 先做一个“反 Chat 症状触发集”，而不是先做大而全的数据集。

如果再加一件第二优先级的事，我会选：

- 立刻开始做 `latest_reminder` 对照实验，因为它很可能正是 recall / belief 注入最自然的 DeepSeek 原生接口。
